# 第十二章：实战 — 手把手写一个新服务

---

> 🎯 **场景模拟：**
> 你刚入职，项目组的微服务架构已经搭好了（就是你现在跑的这个项目）。
> PM 来了一个新需求："我们需要一个**商品评价服务**，用户下单后可以给商品写评价。"
>
> 现在轮到你来写这个 `review-service`。
> 这章不会一次给你看完整代码——我们**一步一步来**，每一步都解释**为什么**。

---

## 第一步：理解需求，设计接口

PM 说了这些：
- 用户购买商品后可以写评价（评分 + 文字）
- 可以查看某个商品的所有评价
- 可以删除自己的评价

**你的思考过程：**

```
需要哪些 API？

POST   /api/v1/reviews          → 创建评价
GET    /api/v1/reviews/{productId} → 查看某商品的所有评价
DELETE /api/v1/reviews/{reviewId}  → 删除评价

需要存什么数据？

Review:
  - id (主键)
  - productId (哪个商品)
  - userId (谁写的)
  - rating (评分 1-5)
  - comment (评价内容)
  - createdAt (创建时间)
```

> 💡 **为什么先想 API 和数据结构？**
> 因为这决定了后面所有代码怎么写。
> Controller 的方法对应 API，Entity 的字段对应数据。
> 先想清楚再写代码，不要上来就敲键盘。

## 第二步：创建项目骨架

模仿 `payment-service` 的结构，创建文件夹：

```
review-service/
├── src/main/java/net/javaguides/review_service/
│   ├── ReviewServiceApplication.java    ← 入口
│   ├── entity/
│   │   └── Review.java                 ← 数据库表
│   ├── repository/
│   │   └── ReviewRepository.java       ← 操作数据库
│   ├── service/
│   │   ├── ReviewService.java          ← 接口
│   │   └── impl/
│   │       └── ReviewServiceImpl.java  ← 实现
│   ├── controller/
│   │   └── ReviewController.java       ← 处理HTTP请求
│   └── dto/
│       └── ReviewRequest.java          ← 接收前端传来的数据
└── src/main/resources/
    └── application.yml                  ← 配置
```

> 💡 **为什么每个服务都是这个结构？**
> 这叫**三层架构**（Controller → Service → Repository）。
> 不是因为"规定这样写"，而是因为它**解决问题**：
> - 如果 API 变了 → 只改 Controller
> - 如果业务逻辑变了 → 只改 Service
> - 如果换数据库 → 只改 Repository
> 每层只干自己的事，改一层不影响其他层。

## 第三步：写入口文件

```java
// ReviewServiceApplication.java

package net.javaguides.review_service;

import org.springframework.boot.SpringApplication;
import org.springframework.boot.autoconfigure.SpringBootApplication;
import org.springframework.cloud.client.discovery.EnableDiscoveryClient;

@SpringBootApplication
@EnableDiscoveryClient
public class ReviewServiceApplication {
    public static void main(String[] args) {
        SpringApplication.run(ReviewServiceApplication.class, args);
    }
}
```

**逐行解释 — 每个注解为什么要写：**

```
@SpringBootApplication
  ↓ 必须写。这个注解等于三合一：
    @Configuration      → "这个类有配置"
    @EnableAutoConfiguration → "自动配置数据库、HTTP服务器等"
    @ComponentScan      → "扫描这个包下所有带 @Controller/@Service 等注解的类"
  ↓ 没有它，Spring 不知道该怎么启动

@EnableDiscoveryClient
  ↓ 可选，但在微服务项目里必须写。
  ↓ 作用：启动时自动把自己注册到 Eureka（服务注册中心）
  ↓ 这样 API Gateway 才能找到你的服务
  ↓ 如果不写 → 网关不知道 review-service 在哪 → 请求转发失败

SpringApplication.run(...)
  ↓ 就是 Java 的 main 方法
  ↓ 启动内置 Tomcat，开始监听 HTTP 请求
```

> 💡 **什么时候需要加注解，什么时候不需要？**
> 规则很简单：
> - `@SpringBootApplication` → **每个服务必须有，只写在 main 类上**
> - `@EnableDiscoveryClient` → **要注册到 Eureka 就加，独立运行可以不加**
> - 你看 payment-service 的 Application 类，也是这两个注解

## 第四步：写 Entity（数据库表）

```java
// entity/Review.java

package net.javaguides.review_service.entity;

import jakarta.persistence.*;
import lombok.AllArgsConstructor;
import lombok.Data;
import lombok.NoArgsConstructor;
import java.time.LocalDateTime;

@Entity                    // ← ①
@Table(name = "reviews")   // ← ②
@Data                      // ← ③
@NoArgsConstructor         // ← ④
@AllArgsConstructor        // ← ⑤
public class Review {

    @Id                                    // ← ⑥
    @GeneratedValue(strategy = GenerationType.UUID)  // ← ⑦
    private String id;

    @Column(nullable = false)              // ← ⑧
    private String productId;

    @Column(nullable = false)
    private String userId;

    @Column(nullable = false)
    private int rating;                    // 1-5 分

    @Column(length = 1000)                 // ← ⑨
    private String comment;

    private LocalDateTime createdAt;

    @PrePersist                            // ← ⑩
    public void prePersist() {
        this.createdAt = LocalDateTime.now();
    }
}
```

**逐个注解解释 — 为什么要写，不写会怎样：**

```
① @Entity
   → 告诉 JPA："这个类对应数据库里的一张表"
   → 不写？Spring 不认识这个类，不会创建表，代码直接报错

② @Table(name = "reviews")
   → 指定表名叫 reviews
   → 不写？默认用类名 Review 作为表名（也能用，但不规范）

③ @Data (Lombok)
   → 自动生成 getter/setter/toString/equals/hashCode
   → 不写？你得手写 20 多行 getter setter
   → 这不是 Spring 的注解，是 Lombok 的（一个代码生成工具）

④ @NoArgsConstructor (Lombok)
   → 生成无参构造方法：public Review() {}
   → 不写会怎样？JPA 要求 Entity 必须有无参构造！没有会报错！
   → JPA 从数据库取数据时，先 new Review() 再一个个 set 字段

⑤ @AllArgsConstructor (Lombok)
   → 生成全参构造方法：public Review(String id, String productId, ...)
   → 不写？也行，但写代码时创建对象不方便

⑥ @Id
   → 标记主键
   → 不写？JPA 不知道哪个字段是主键，直接报错

⑦ @GeneratedValue(strategy = GenerationType.UUID)
   → 主键自动生成，用 UUID（一串随机字符串）
   → 还有其他策略：GenerationType.IDENTITY（自增数字 1,2,3...）
   → 项目里用 UUID 因为微服务各自的数据库，自增 ID 会冲突

⑧ @Column(nullable = false)
   → 这个字段在数据库里不能为 NULL
   → 不写？默认允许 NULL，但 productId 为空没意义

⑨ @Column(length = 1000)
   → 字符串最长 1000 字符（默认 255）
   → 评价内容可能比较长，所以加大

⑩ @PrePersist
   → "在保存到数据库之前自动执行"
   → 这样 createdAt 不需要手动设置，保存时自动填入当前时间
```

> 💡 **对比项目里的 Payment Entity**
> 打开 `payment-service/entity/Payment.java` 对比一下，
> 你会发现注解几乎一样——因为 Entity 的写法是固定模式。

## 第五步：写 Repository（操作数据库）

```java
// repository/ReviewRepository.java

package net.javaguides.review_service.repository;

import net.javaguides.review_service.entity.Review;
import org.springframework.data.jpa.repository.JpaRepository;
import java.util.List;

public interface ReviewRepository extends JpaRepository<Review, String> {

    List<Review> findByProductId(String productId);

    List<Review> findByUserId(String userId);
}
```

**你可能的疑问：**

```
Q: 为什么是 interface 不是 class？
A: 因为 JPA 帮你实现了！你只需要定义"我想要什么方法"，
   Spring Data JPA 在运行时自动生成实现代码。
   你不需要写任何 SQL！

Q: extends JpaRepository<Review, String> 是什么意思？
A: JpaRepository 是 Spring 提供的，有很多现成方法：
   - save(entity)     → INSERT 或 UPDATE
   - findById(id)     → SELECT * WHERE id = ?
   - findAll()        → SELECT *
   - deleteById(id)   → DELETE WHERE id = ?
   <Review, String> 的意思是：
   - Review → 操作哪个 Entity
   - String → 主键的类型（Review 的 id 是 String）

Q: findByProductId 没有写实现，它怎么知道要查什么？
A: Spring Data JPA 根据方法名自动生成 SQL！
   findByProductId → SELECT * FROM reviews WHERE product_id = ?
   findByUserId    → SELECT * FROM reviews WHERE user_id = ?
   这叫 "方法名查询"，命名规则是 findBy + 字段名

Q: 为什么没有写 @Repository 注解？
A: 继承 JpaRepository 后 Spring 自动识别，不需要手动加。
   但如果你写了一个普通的 DAO 类（不继承 JpaRepository），
   就需要加 @Repository。
```

> 💡 **对比项目里的 PaymentRepository**
> 打开 `payment-service/repository/PaymentRepository.java`，
> 写法一模一样：interface + extends JpaRepository。

## 第六步：写 DTO（传输对象）

```java
// dto/ReviewRequest.java

package net.javaguides.review_service.dto;

import lombok.Data;

@Data
public class ReviewRequest {
    private String productId;
    private String userId;
    private int rating;
    private String comment;
}
```

**为什么不直接用 Entity 接收前端数据？**

```
Entity (Review):
  - id           ← 自动生成，前端不该传
  - productId    ← 前端传
  - userId       ← 前端传
  - rating       ← 前端传
  - comment      ← 前端传
  - createdAt    ← 自动生成，前端不该传

如果用 Entity 接收，前端可以偷偷传 id 和 createdAt，
篡改你的数据！

DTO 只包含"前端应该传的字段"，是一层安全过滤。

面试话术："Entity 直接对应数据库表，
DTO 是面向接口的数据传输对象，
两者职责不同，不应混用。"
```

## 第七步：写 Service（业务逻辑）

**先写接口（定义"能做什么"）：**

```java
// service/ReviewService.java

package net.javaguides.review_service.service;

import net.javaguides.review_service.dto.ReviewRequest;
import net.javaguides.review_service.entity.Review;
import java.util.List;

public interface ReviewService {
    Review createReview(ReviewRequest request);
    List<Review> getReviewsByProductId(String productId);
    void deleteReview(String reviewId);
}
```

**再写实现（定义"怎么做"）：**

```java
// service/impl/ReviewServiceImpl.java

package net.javaguides.review_service.service.impl;

import lombok.RequiredArgsConstructor;               // ← Lombok
import net.javaguides.review_service.dto.ReviewRequest;
import net.javaguides.review_service.entity.Review;
import net.javaguides.review_service.repository.ReviewRepository;
import net.javaguides.review_service.service.ReviewService;
import org.springframework.stereotype.Service;

@Service                      // ← ①
@RequiredArgsConstructor      // ← ②
public class ReviewServiceImpl implements ReviewService {

    private final ReviewRepository reviewRepository;  // ← ③

    @Override
    public Review createReview(ReviewRequest request) {
        // 把 DTO 转成 Entity
        Review review = new Review();
        review.setProductId(request.getProductId());
        review.setUserId(request.getUserId());
        review.setRating(request.getRating());
        review.setComment(request.getComment());

        // 保存到数据库
        return reviewRepository.save(review);
        // save() 是 JpaRepository 自带的方法
        // 自动执行 INSERT INTO reviews VALUES (...)
    }

    @Override
    public List<Review> getReviewsByProductId(String productId) {
        return reviewRepository.findByProductId(productId);
        // 自动执行 SELECT * FROM reviews WHERE product_id = ?
    }

    @Override
    public void deleteReview(String reviewId) {
        reviewRepository.deleteById(reviewId);
        // 自动执行 DELETE FROM reviews WHERE id = ?
    }
}
```

**注解解释：**

```
① @Service
   → 告诉 Spring："这个类是业务逻辑层，请管理它"
   → Spring 启动时自动创建这个类的实例，放进容器
   → 不写？Spring 不认识它 → Controller 注入时找不到 → 启动报错

② @RequiredArgsConstructor (Lombok)
   → 自动生成构造方法，参数是所有 final 字段
   → 等效于手写：
     public ReviewServiceImpl(ReviewRepository reviewRepository) {
         this.reviewRepository = reviewRepository;
     }
   → Spring 看到构造方法需要 ReviewRepository → 自动注入

③ private final ReviewRepository reviewRepository
   → private: 封装，外部不能直接访问
   → final: 不可变，一旦注入就不能换
   → 这就是"构造器注入"
```

> 💡 **为什么要拆成接口 + 实现？上面第一章讲过，这里再强调：**
> - Controller 里写 `private final ReviewService reviewService`
> - 它**不知道**具体实现是 ReviewServiceImpl
> - 以后换实现（比如加缓存版本），Controller 不用改
> - 测试时可以 Mock 一个假的 ReviewService

## 第八步：写 Controller（处理 HTTP 请求）

```java
// controller/ReviewController.java

package net.javaguides.review_service.controller;

import lombok.RequiredArgsConstructor;
import net.javaguides.review_service.dto.ReviewRequest;
import net.javaguides.review_service.entity.Review;
import net.javaguides.review_service.service.ReviewService;
import org.springframework.http.HttpStatus;
import org.springframework.http.ResponseEntity;
import org.springframework.web.bind.annotation.*;

import java.util.List;

@RestController                           // ← ①
@RequestMapping("/api/v1/reviews")        // ← ②
@RequiredArgsConstructor                  // ← ③
public class ReviewController {

    private final ReviewService reviewService;  // ← ④ 注入接口，不是实现

    @PostMapping                          // ← ⑤
    public ResponseEntity<Review> createReview(
            @RequestBody ReviewRequest request) {  // ← ⑥
        Review review = reviewService.createReview(request);
        return ResponseEntity
                .status(HttpStatus.CREATED)    // 返回 201 状态码
                .body(review);                 // 返回创建的 Review
    }

    @GetMapping("/{productId}")           // ← ⑦
    public ResponseEntity<List<Review>> getReviews(
            @PathVariable String productId) {  // ← ⑧
        return ResponseEntity.ok(
                reviewService.getReviewsByProductId(productId));
    }

    @DeleteMapping("/{reviewId}")         // ← ⑨
    public ResponseEntity<String> deleteReview(
            @PathVariable String reviewId) {
        reviewService.deleteReview(reviewId);
        return ResponseEntity.ok("Review deleted");
    }
}
```

**逐个注解解释：**

```
① @RestController
   = @Controller + @ResponseBody
   → @Controller: "这个类处理 HTTP 请求"
   → @ResponseBody: "方法返回值直接作为 HTTP 响应体（JSON）"
   → 不写？Spring 不知道这个类要处理请求 → 访问时返回 404

② @RequestMapping("/api/v1/reviews")
   → 这个 Controller 处理所有 /api/v1/reviews 开头的请求
   → 为什么路径是 /api/v1/？这是 RESTful 规范：
     /api   → 表示是 API 接口
     /v1    → 版本号，以后改接口可以出 /v2 而不影响旧版
     /reviews → 资源名，用复数

③ @RequiredArgsConstructor
   → 自动注入 reviewService（和 Service 层一样的原理）

④ private final ReviewService reviewService
   → 注入的是接口！不是 ReviewServiceImpl！
   → Spring 自动找到 @Service 标注的实现类注入进来

⑤ @PostMapping
   → 处理 POST 请求（创建资源用 POST）
   → 完整路径 = @RequestMapping + @PostMapping = /api/v1/reviews

⑥ @RequestBody ReviewRequest request
   → 把前端发来的 JSON body 自动转成 ReviewRequest 对象
   → 前端发 {"productId":"xxx","rating":5,"comment":"很好"}
   → Spring 自动变成 request.getProductId() = "xxx"

⑦ @GetMapping("/{productId}")
   → 处理 GET 请求到 /api/v1/reviews/{productId}
   → {productId} 是路径变量，值自动传给方法参数

⑧ @PathVariable String productId
   → 从 URL 取值：/api/v1/reviews/prod-001 → productId = "prod-001"

⑨ @DeleteMapping("/{reviewId}")
   → 处理 DELETE 请求（删除资源用 DELETE）
```

**返回值为什么用 `ResponseEntity<Review>` 而不是 `ResponseEntity<?>`？**

```
ResponseEntity<Review>
  → 明确告诉调用方："我返回的一定是 Review 对象"
  → 更规范，IDE 能自动提示，Swagger 文档更清晰

ResponseEntity<?>
  → "我可能返回不同类型的东西"
  → 项目里用 <?> 是因为同时返回成功（对象）和失败（错误字符串）
  → 更好的做法是统一包装成 ApiResponse<T>，项目里也有这样做

面试答法：
"我倾向于用具体类型 ResponseEntity<T> 而不是通配符 <?>，
因为它类型安全，文档也更清晰。
如果需要统一返回格式，我会用 ApiResponse<T> 包装。"
```

## 第九步：写配置文件

```yaml
# application.yml

server:
  port: 8086                # 每个服务用不同端口

spring:
  application:
    name: REVIEW-SERVICE    # 注册到 Eureka 的服务名（大写）

  datasource:
    url: jdbc:mysql://localhost:3312/review_db  # 独立数据库
    username: root
    password: root
    driver-class-name: com.mysql.cj.jdbc.Driver

  jpa:
    hibernate:
      ddl-auto: update     # 自动建表（生产环境用 validate）
    show-sql: true          # 控制台打印 SQL（调试用）

eureka:
  client:
    service-url:
      defaultZone: http://localhost:8761/eureka/  # Eureka 地址
```

**每个配置为什么这样写：**

```
server.port: 8086
  → 项目里的端口：8081(product), 8082(order), 8083(payment),
    8084(email), 8085(identity)
  → 新服务就用 8086，不能和现有的冲突

spring.application.name: REVIEW-SERVICE
  → 注册到 Eureka 的名字
  → API Gateway 根据这个名字转发请求
  → 必须大写（Eureka 的惯例）

spring.datasource
  → 微服务原则：每个服务有自己的数据库
  → review-service 用 review_db，不和别的服务共享
  → 端口 3312 是因为 docker-compose 里每个 MySQL 映射不同端口

jpa.hibernate.ddl-auto: update
  → 启动时自动检查 Entity 和数据库表的差异，自动修改
  → 开发时用 update（方便），生产时用 validate（只检查不改）
```

## 第十步：在 API Gateway 加路由

打开 `api-gateway/src/main/resources/application.yml`，加一段：

```yaml
# 在 routes 列表里加上这个：
- id: review-service
  uri: lb://REVIEW-SERVICE         # lb = load balance，从 Eureka 找
  predicates:
    - Path=/api/v1/reviews/**      # 匹配 /api/v1/reviews 开头的请求
```

**这样加完后：**

```
用户请求 → http://localhost:9191/api/v1/reviews/prod-001
         ↓
API Gateway (9191) 看路径匹配 /api/v1/reviews/**
         ↓
问 Eureka：REVIEW-SERVICE 在哪？→ localhost:8086
         ↓
转发到 review-service (8086)
         ↓
ReviewController.getReviews("prod-001")
         ↓
返回评价列表 JSON
```

## 总结：你刚才做了什么

```
你写了 6 个文件：
  1. ReviewServiceApplication.java  ← 入口（2个注解）
  2. Review.java                    ← Entity（6个注解）

  5. ReviewServiceImpl.java         ← 业务逻辑（2个注解）
  6. ReviewController.java          ← HTTP接口（4个注解）

+ 1 个配置文件：application.yml
+ 1 行路由配置：api-gateway 的 routes

总代码量：大约 150 行
花的时间：跟着做大约 30 分钟

这就是微服务开发的日常——
每个新功能就是一个新服务，结构几乎一样，
区别只在业务逻辑（Service 层）不同。
```

## 注解速查表（随时翻）

| 注解 | 来自 | 用在哪 | 作用 | 必须写？ |
|------|------|--------|------|---------|
| `@SpringBootApplication` | Spring Boot | Application 类 | 启动 Spring 容器 | ✅ 必须 |
| `@EnableDiscoveryClient` | Spring Cloud | Application 类 | 注册到 Eureka | 微服务必须 |
| `@Entity` | JPA | Entity 类 | 映射数据库表 | ✅ 必须 |
| `@Table(name="xxx")` | JPA | Entity 类 | 指定表名 | 不写用类名 |

```sql
-- 从1万条快速翻到100万条（每次翻倍，比逐条INSERT快100倍）
INSERT INTO reviews (id, name, description, created_at)
SELECT UUID(), CONCAT('R-', FLOOR(RAND()*9999999)),

  DATE_SUB(NOW(), INTERVAL FLOOR(RAND()*730) DAY)
```

## 第十四步：优化 #1 — 分页（Pagination）

### 代码改动

**Controller 加 page/size 参数：**

```java
@GetMapping
public ResponseEntity<Page<Review>> getAll(
        @RequestParam(defaultValue = "0") int page,     // 第几页
        @RequestParam(defaultValue = "20") int size) {   // 每页多少条

    if (size > 100) size = 100;  // 防止恶意请求

    PageRequest pageRequest = PageRequest.of(page, size,
            Sort.by(Sort.Direction.DESC, "createdAt"));  // 最新的在前面
    
    return ResponseEntity.ok(reviewService.getAll(pageRequest));
}

// 用法：
// GET /api/v1/reviews?page=0&size=20     → 第1页，每页20条
// GET /api/v1/reviews?page=5&size=50     → 第6页，每页50条
```

**Service 改用 Pageable：**

```java
Page<Review> getAll(Pageable pageable);
```

**Repository 不用改 — JpaRepository 自带分页：**

```java
// findAll(Pageable) 是 JpaRepository 内置方法
// 自动生成 SQL: SELECT * FROM reviews ORDER BY created_at DESC LIMIT 20 OFFSET 0
```

### 分页后压测：

```
Requests/sec:     337.77   ← ✅ 提升 16 倍！
Avg Latency:      293ms    ← ✅ 快了 4 倍
Timeout:          0        ← ✅ 完全消除
Transfer/sec:     1.25MB   ← ✅ 数据量减少 28 倍
```

> 💡 **生产铁律：**
> - ❌ 永远不要 `findAll()` 无条件返回全部数据
> - ✅ 所有列表接口**必须分页**
> - ✅ 限制 `size` 最大值（防止 `?size=999999`）
> - ✅ 默认按时间倒序（用户看最新的）

## 第十五步：优化 #2 — 数据库索引（Index）


## 第十六步：优化 #3 — Slice 替换 Page（去掉 COUNT）

看 Hibernate 日志发现了罪魁祸首：

```sql
-- 每次请求实际执行了 2 条 SQL：
SELECT * FROM reviews ORDER BY created_at DESC LIMIT 20;      -- ✅ 很快
SELECT COUNT(*) FROM reviews;                                    -- 💀 超慢！
```

**Page 需要 COUNT 来算总页数，但 100 万条数据的 COUNT 要全表扫描！**

### 解法：用 Slice 代替 Page

```java
// Repository 里加一个方法：
public interface ReviewRepository extends JpaRepository<Review, String> {
    // 返回 Slice 而不是 Page → 不执行 COUNT(*)
    Slice<Review> findAllBy(Pageable pageable);
}

// Controller 改用 Slice
@GetMapping
public ResponseEntity<Slice<Review>> getAll(...) {
    return ResponseEntity.ok(reviewService.getAllSlice(pageRequest));
}
```

**Page vs Slice 区别：**

```
Page:
  → 执行 2 条 SQL: SELECT + COUNT
  → 返回: 数据 + totalElements + totalPages
  → 适合: 数据量小、需要显示"共X页"的场景

Slice:
  → 执行 1 条 SQL: SELECT (多取1条判断有没有下一页)
  → 返回: 数据 + hasNext (true/false)
  → 适合: 数据量大、无限滚动(像朋友圈)的场景

京东商品列表用的就是 Slice 思路——
你翻到底部自动加载下一页，而不是显示"共999页"
```

### 最终压测结果（100 万数据）：

```
50  并发:  1,490 req/s    Latency 50ms     0 timeout
200 并发:  2,469 req/s    Latency 101ms    0 timeout
500 并发:  2,432 req/s    Latency 250ms    0 timeout（开始触顶）
```

## 性能优化总结

```
从"玩具"到"生产级"只做了 3 件事：

1. 分页（Pagination） → 不返回全部数据
2. 数据库索引（Index） → 加速 ORDER BY 和 WHERE

这不是什么黑科技——这就是生产环境的基本功。
```
