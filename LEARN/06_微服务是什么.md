## 📖 Part 1：从你懂的 Java 出发

### 单体应用：所有代码放一起

```
想象你刚学 Java 做了一个电商系统，代码结构：

MyShopApp/
├── UserModule.java      ← 用户登录注册
├── ProductModule.java   ← 商品管理
├── OrderModule.java     ← 下单
├── PaymentModule.java   ← 支付
├── EmailModule.java     ← 发邮件
└── main.java            ← 所有功能在一个程序里运行

这叫"单体应用（Monolith）"
```

**单体的优点：**
- 开发简单（一个项目就行）
- 部署简单（打一个包就行）
- 调试简单（什么都在一个进程里）

**单体的问题（公司大了之后）：**

```
问题1：一个模块出 bug → 整个系统挂掉
  支付模块内存泄漏 → 用户登录、浏览商品全部崩溃

问题2：改一行代码 → 重新部署整个系统
  改了订单模块里一个小 bug → 整个系统要重新发布
  万一发布出问题 → 所有功能都受影响

问题3：不同模块要不同的资源
  双11，订单模块压力巨大 → 需要 100 台服务器
  邮件模块毫无压力 → 只需要 1 台服务器
  但它们是同一个程序 → 所有 100 台都跑着完整的系统，浪费！

问题4：不同团队改同一份代码
  订单团队改 OrderModule → 不小心影响了 PaymentModule
  10个人改同一个程序 → 频繁冲突
```

---

### 微服务：每个功能独立运行

```
你现在的项目就是微服务！

identity-service/   ← 独立的程序1，管用户（端口 8085）
product-service/    ← 独立的程序2，管商品（端口 8081）
order-service/      ← 独立的程序3，管订单（端口 8082）
payment-service/    ← 独立的程序4，管支付（端口 8083）
email-service/      ← 独立的程序5，管邮件（端口 8084）

每个程序：
  ✅ 独立运行（各有各的 main 方法）
  ✅ 独立数据库（各有各的 MySQL）
  ✅ 独立部署（改订单不影响支付）
  ✅ 独立扩容（订单压力大就多开几个 order-service）
```

## 💻 Part 2：在这个项目里怎么用？

### 问题一：服务之间怎么找到对方？
### → Eureka 服务注册中心

```
没有 Eureka 时：
  order-service 要调用 payment-service
  order-service 写死地址：http://192.168.1.100:8083/...
  
  问题：payment-service 换了机器怎么办？地址都要改！
        payment-service 开了 3 个实例，怎么知道调哪个？

有了 Eureka：
  1. 每个服务启动时，告诉 Eureka："我是 xxx-service，我的地址是 yyy"
  2. order-service 想找 payment-service 时
     → 问 Eureka："payment-service 在哪？"
     → Eureka 回答："它有 3 个实例，地址分别是..."
     → order-service 选一个调用（负载均衡）
```

```
路径：service-registry/src/main/java/.../ServiceRegistryApplication.java
```

```java
@SpringBootApplication
@EnableEurekaServer     // ← 一个注解就开启了注册中心！
public class ServiceRegistryApplication {
    public static void main(String[] args) {
        SpringApplication.run(ServiceRegistryApplication.class, args);
    }
}
```

**其他服务怎么注册？**

```yaml
# 每个服务的 application.yml 都有这段配置：
eureka:
  client:
    service-url:
      defaultZone: http://eureka-server:8761/eureka/
  #                  ↑ Eureka 的地址
  instance:
    prefer-ip-address: true

spring:
  application:
    name: payment-service    # ← 注册名，其他服务用这个名字找它
```

```
启动后打开 http://localhost:8761 能看到所有注册的服务：

Application          Status
IDENTITY-SERVICE     UP (1)
PRODUCT-SERVICE      UP (1)
ORDER-SERVICE        UP (1)
PAYMENT-SERVICE      UP (1)
EMAIL-SERVICE        UP (1)
API-GATEWAY          UP (1)
```

### 问题二：服务之间怎么互相调用？
### → OpenFeign（声明式 HTTP 客户端）

```
order-service 需要查商品价格 → 要调用 product-service 的 API
```

**不用 Feign 时（手动 HTTP 调用）：**
```java
// 手动用 RestTemplate
RestTemplate restTemplate = new RestTemplate();
Product product = restTemplate.getForObject(
    "http://product-service:8081/api/v1/products/" + productId,
    Product.class
);
// 要手动拼 URL、处理序列化、处理异常...
```

**用 Feign 时（像调本地方法一样调远程服务）：**
```java
// 路径：order-service/.../service/ProductAPIClient.java

@FeignClient(name = "product-service")  // 要调的服务名（在 Eureka 注册的名字）
public interface ProductAPIClient {

    @GetMapping("/api/v1/products/{productId}")
    ProductDTO getProductById(@PathVariable String productId);
    //  ↑ 看起来就像普通的接口方法！
    //    但 Feign 会自动：
    //    1. 去 Eureka 找 product-service 的地址
    //    2. 拼出完整的 URL
    //    3. 发 HTTP GET 请求
    //    4. 把 JSON 响应转成 ProductDTO 对象
    //    5. 返回给你
}

// 在 Service 里直接像本地方法一样用：
@Service
@RequiredArgsConstructor
public class OrderServiceImpl {
    
    private final ProductAPIClient productClient;  // 注入 Feign 客户端
    
    public void createOrder(OrderDTO orderDTO) {
        // 像调本地方法一样调远程服务！
        ProductDTO product = productClient.getProductById("prod-001");
        BigDecimal price = product.getPrice();
        // ...
    }
}
```

### 问题三：所有请求怎么统一入口？
### → API Gateway（网关）

```
没有网关时：
  前端要记住每个服务的端口：
  - 登录 → http://localhost:8085/api/v1/auth/token
  - 查商品 → http://localhost:8081/api/v1/products
  - 下单 → http://localhost:8082/api/v1/orders
  
  问题：5个服务5个端口！前端疯了！

有了网关：
  前端只需要记一个地址：http://localhost:9191
  网关帮你转发：
  - /api/v1/auth/**    → identity-service
  - /api/v1/products/** → product-service
  - /api/v1/orders/**   → order-service
```

```yaml
# 路径：api-gateway/src/main/resources/application.yml

spring:
  cloud:
    gateway:
      routes:
        - id: identity-service
          uri: lb://identity-service      # lb:// = 负载均衡
          predicates:
            - Path=/api/v1/auth/**        # 路径匹配

        - id: product-service
          uri: lb://product-service
          predicates:
            - Path=/api/v1/products/**

        - id: order-service
          uri: lb://order-service
          predicates:
            - Path=/api/v1/orders/**
```

**API Gateway 还做了什么？**
- ✅ JWT 验证（所有请求先验 Token，无效直接拒绝）
- ✅ 负载均衡（同一个服务有多个实例，轮流调用）
- ✅ 统一错误处理
- ✅ 日志记录

---

### 完整调用链路

```
用户浏览器
    ↓ POST /api/v1/orders （带 JWT Token）
API Gateway (9191)
    ↓ ① 验证 JWT → 合法
    ↓ ② 路径匹配 /api/v1/orders → 转发到 order-service
    ↓ ③ 问 Eureka：order-service 在哪？→ 得到 192.168.1.10:8082
    ↓
order-service (8082)
    ↓ ④ 通过 Feign 调用 product-service 查商品价格
    ↓ ⑤ 创建订单存入 MySQL
    ↓ ⑥ 发 Kafka 消息通知 payment-service
    ↓
返回给用户："订单创建成功"
```

## 🎤 Part 3：面试官会怎么问？

---

**Q: 你为什么用微服务架构？**

> 🗣️ "这个项目是电商系统，业务上可以清晰拆分为商品、订单、支付、
> 邮件、身份认证这些独立模块。微服务让每个模块独立部署、
> 独立扩容。比如大促时订单压力大，可以只扩 order-service 的实例，
> 不影响其他服务。如果是个小项目或一两个人的团队，我会选择单体应用。"

---

**Q: 微服务有什么缺点？**

> 🗣️ "主要三个缺点：
> 第一，分布式事务复杂，一个操作跨多个服务时很难保证数据一致性；
> 第二，运维成本高，7个服务意味着7套部署、监控、日志；
> 第三，网络调用有延迟和失败风险，需要做超时、重试、熔断处理。"

---

**Q: 服务之间怎么通信的？**

> 🗣️ "两种方式。同步通信用 OpenFeign，比如 order-service 查商品价格时
> 需要实时返回结果，就同步调用 product-service。
> 异步通信用 Kafka，比如下单成功后发消息给 payment-service 
> 和 email-service，不需要等它们处理完。"

---

**Q: Eureka 是什么？为什么需要它？**

> 🗣️ "Eureka 是服务注册与发现中心。每个服务启动时向 Eureka 注册
> 自己的地址，其他服务需要调用时从 Eureka 查询地址。
> 这样服务的地址变化时不需要改配置，而且多实例时 
> 可以通过 Eureka 实现客户端负载均衡。"
